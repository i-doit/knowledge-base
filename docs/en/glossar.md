# Glossary

IT speaks its own language. This glossary helps you understand what lies behind the terms in i-doit. Here you will find the most important technical terms related to the CMDB, object types, categories, and other concepts in i-doit -- sorted alphabetically with brief explanations.

<!-- cSpell:disable -->
| Term                                                                          | Explanation                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **General Category**{#allgemein-kategorie}                                       | The "General" category belongs to those [categories](#kategorie) that are an integral part of every [object](#objekt). Therefore, it cannot be deselected per [object type](#objekttyp). This category records important [attributes](#attribut) such as the [object title](#objekt-titel), [SYS-ID](#sys-id), [CMDB status](#cmdb-status), and [condition](#zustand). |
| **Attribute**{#attribut}                                                         | An attribute is a documented value of an [object](#objekt). Similar attributes are grouped into [categories](#kategorie) in i-doit. Example: the [CMDB status](#cmdb-status) in the "[General](#allgemein-kategorie)" category. |
| **Attribute Field**{#attributfeld}                                               | In the i-doit web GUI, [attributes](#attribut) of [objects](#objekt) are saved and edited via form fields. These fields are therefore called attribute fields. They can be of various types: single-line or multi-line text fields, date fields, HTML editors, [object browsers](#objekt-browser), [Dialog-Plus fields](#dialog-plus-feld), and many more. |
| **Custom Category**{#benutzerdefinierte-kategorie}                               | A [category](#kategorie) that has been created by a user and configured with one or more [attribute fields](#attributfeld) is marked as custom in i-doit. A corresponding article provides [instructions for creation](grundlagen/benutzerdefinierte-kategorien.md). |
| **Configuration Management Database (CMDB)**{#configuration-management-database} | A Configuration Management Database, or CMDB, allows the management of and access to Configuration Items (CI). As software for [IT documentation](#it-dokumentation), i-doit covers important functionalities of a CMDB. [More information about CMDBs is available on Wikipedia.](https://en.wikipedia.org/wiki/Configuration_management_database) |
| **CMDB Status**{#cmdb-status}                                                    | To document the lifecycle of an [object](#objekt), such as a server or software, i-doit uses the CMDB status. This is documented per object as an [attribute](#attribut) of the "[General](#allgemein-kategorie)" [category](#kategorie). In the default state of i-doit, it includes important stages in a lifecycle. Examples are "In operation", "Planned", "Defective". See also [Condition](#zustand). |
| **Dialog-Plus Field**{#dialog-plus-feld}                                         | A Dialog-Plus field is an [attribute field](#attributfeld) for selecting a preconfigured value. In web terminology, it is called a dropdown field. Any number of additional values can be added or edited. In some cases, i-doit already comes with preconfigured Dialog-Plus fields. |
| **Extension**{#erweiterung}                                                      | Extensions are available free of charge and provide additional functions for your documentation. They can be downloaded via the [customer portal](administration/kundenportal.md). installation is done separately via the i-doit [Admin Center](administration/admin-center.md). |
| **Field**{#feld}                                                                 | See [Attribute Field](#attributfeld) |
| **Global Category**{#globale-kategorie}                                          | A global [category](#kategorie) can in principle be assigned to any [object type](#objekttyp), so that its [attributes](#attribut) are available to the respective [objects](#objekt). |
| **i-doit**{#i-doit}                                                              | The name i-doit is a play on words and originally stands for "I document IT". And yes, we know the joke with i-diot! |
| **IT Documentation**{#it-dokumentation}                                          | IT documentation encompasses technical documentation, administrative management, consolidation of ITSM data, and modeling of services. The technical part includes the configuration of [objects](#objekt) and their [relationships](#objektbeziehung) to each other. i-doit provides a multidimensional data model for this purpose, to collect, process, and reuse data in different contexts in a structured manner. IT documentation is to be understood as a discipline of IT Service Management (ITSM), which plays a significant role in every IT organization. The terms "IT documentation" and "CMDB" are often used interchangeably. |
| **Category**{#kategorie}                                                         | Thematically related [attributes](#attribut) are grouped into categories per [object](#objekt). There are three types of categories: [global](#globale-kategorie), [specific](#spezifische-kategorie), and [custom](#benutzerdefinierte-kategorie). They are further divided into [single-value](#single-value-kategorie) and [multi-value categories](#multi-value-kategorie). Some categories are [backward](#rueckwaertige-kateogorie), while others serve as [views](#view-kategorie). Finally, there are special categories like "[General](#allgemein-kategorie)" or the "[Overview Page](#uebersichtsseite)". |
| **Category Folder**{#kategorieordner}                                            | Some thematically related [categories](#kategorie) are grouped into a folder structure. An example is the "Network" category folder, which contains the documentation of physical and logical ports as well as interfaces in dedicated subcategories. This folder structure cannot be edited, nor can only subcategories be configured per [object type](#objekttyp). |
| **List Category**{#listenkategorie}                                              | See [Multi-Value Category](#multi-value-kategorie) |
| **Multi-Value Category**{#multi-value-kategorie}                                 | When the attributes of a [category](#kategorie) can be documented multiple times per [object](#objekt), it is called a multi-value category. An example is the "CPU" category: for a multi-socket system, each CPU can be documented separately with frequency, core count, etc. The counterpart is the [single-value category](#single-value-kategorie). |
| **Module**{#modul}                                                               | Modules are paid [extensions](#erweiterung) with a significantly larger feature set. They are provided after purchase and can be installed via the i-doit [Admin Center](administration/admin-center.md). |
| **Object**{#objekt}                                                              | Objects in i-doit are all things documented in [IT documentation](#it-dokumentation), whether physical devices like servers or clients, or logical constructs like networks or services. An object is defined by its [object type](#objekttyp), which determines which [attributes](#attribut) can be populated with values. In ITIL, the term "Configuration Item (CI)" is used. In asset management, the term "asset" is common. The term "object" aims to establish a more abstract and thus universally applicable definition. |
| **Object Browser**{#objekt-browser}                                              | When [objects](#objekt) are in a [relationship](#objektbeziehung), this relationship is often documented in i-doit via an object browser. This is an [attribute field](#attributfeld) used in many [categories](#kategorie). An example is the [attribute](#attribut) "Purchased from" in the "Accounting" category, to document from whom an object was purchased. |
| **Object Relationship**{#objektbeziehung}                                        | In i-doit, IT components can not only be documented standalone but also be put into relationships. There are various preconfigured relationship types that can be customized and extended. Each relationship is a separate (non-licensed) [object](#objekt) that is automatically created, edited, or deleted. A relationship consists of a master and a slave object. Additionally, a weighting can be assigned. |
| **Object List**{#objektliste}                                                    | Any number of [objects](#objekt) can be stored for each [object type](#objekttyp). These are displayed in i-doit in the object list as a table. The columns and which [attributes](#attribut) they display can be configured. |
| **Object Title**{#objekt-titel}                                                  | Every [object](#objekt) in i-doit receives a title. This is documented as an [attribute](#attribut) in the "[General](#allgemein-kategorie)" [category](#kategorie). This attribute is also synonymously called "designation" or "name". |
| **Object Type**{#objekttyp}                                                      | An object type is the grouping of all [objects](#objekt) of the same type. Examples are "Router", "Server", or "Applications". This grouping is commonly also referred to as "class". In ITIL, the term "CI Type" is used. |
| **Object Type Group**{#objekttypgruppe}                                          | In a well-maintained and well-populated [IT documentation](#it-dokumentation), many [object types](#objekttyp) are often in use. To present this variety in a clear manner, similar object types can be grouped. These groups appear in the main navigation bar (top area) of i-doit. |
| **Quicklaunch Widget**{#quicklaunch-widget}                                      | The Quicklaunch widget enables quick access to frequently used functions in i-doit. These can be accessed directly via the widget. |
| **Root Location**{#root-lokation}                                                | This "virtual" location is needed to assign a location to those locations that are at the top of the location hierarchy. |
| **Backward Category**{#rueckwaertige-kateogorie}                                 | When two or more [objects](#objekt) are in a [relationship](#objektbeziehung) with each other, this is recorded in the designated [category](#kategorie). For example, in the "Contact assignment" category, any number of persons, person groups, etc. can be assigned as contacts for an object. So that it is also visible within these persons and person groups which objects they are assigned to as contacts, an additional category "Assigned Objects" exists. Since the same information is available there but is only stored once and displayed in a different context, it is a backward category. |
| **Single-Value Category**{#single-value-kategorie}                               | In a single-value category, each associated [attribute](#attribut) can only be documented once per [object](#objekt). An example is the [category](#kategorie) "Accounting": information about inventory number, cost center, etc. is only needed once. The counterpart is the [multi-value category](#multi-value-kategorie). |
| **Specific Category**{#spezifische-kategorie}                                    | A specific category differs from a [global category](#globale-kategorie) in that its [attributes](#attribut) have a very specific focus that is typically suitable for only one or a few [object types](#objekttyp). An example is the [category](#kategorie) "Rack", which is assigned to the object type of the same name "Rack" and presents a rack view to the user. |
| **SYS-ID**{#sys-id}                                                              | SYS-ID stands for "System Identifier" and automatically assigns a unique number per [object](#objekt). It is stored as an [attribute](#attribut) in the "[General](#allgemein-kategorie)" [category](#kategorie). In the default state of i-doit, the SYS-ID consists of the prefix "SYSID\_" and a sequence of digits. In the object type configuration, a prefix can be assigned per [object type](#objekttyp), for example "SRV\_" for servers. The [attribute field](#attributfeld) for the SYS-ID can be marked as read-only under **Administration > [Tenant Name] Administration > CMDB**. |
| **Overview Page**{#uebersichtsseite}                                             | The overview page is displayed when opening an [object](#objekt) in the i-doit web GUI. It consists of the "[General](#allgemein-kategorie)" [category](#kategorie) and optionally additional categories assigned to the [object type](#objekttyp). These can be selected/deselected and sorted in the object type configuration. |
| **View Category**{#view-kategorie}                                               | In some [categories](#kategorie), no [attributes](#attribut) can be stored per [object](#objekt). They serve to evaluate data stored elsewhere. The processed data cannot be edited. An example is the "Object Vitality" category, an evaluation of the "CPU", "Memory", "Port", and "Software Assignment" categories. |
| **Condition**{#zustand}                                                          | To describe the lifecycle of [IT documentation](#it-dokumentation), the [attribute](#attribut) "Condition" is used, which is recorded in the "[General](#allgemein-kategorie)" [category](#kategorie) per [object](#objekt). Every newly created object receives the condition "normal". When the documentation of an object is no longer needed, it can be "archived". The next step in the lifecycle is the condition "deleted". Up to this point, the documented data remains completely preserved in the IT documentation but is hidden. Only the final step "purge" irrevocably deletes the documentation of an object including its associated [object relationships](#objektbeziehung) from i-doit. See also [CMDB Status](#cmdb-status). |
